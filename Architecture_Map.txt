--- APP OVERVIEW ---

A construction workforce management platform built around a Kanban-style order
(zayavka) system. Foremen create work orders assigning teams, equipment, and
schedules to construction sites. Admins review/approve orders, which are then
auto-published as visual schedule images to Telegram and MAX group chats.

Key capabilities:
  - Kanban order workflow (pending -> approved -> published -> in_progress -> completed)
  - Time-based equipment scheduling with free-slot detection
  - Equipment exchange between foremen (Stage 5A)
  - Account linking: merge Telegram <-> MAX identities (Stage 5B)
  - Cost estimation (KP) per order and per construction site
  - Team/brigade management with invite codes and passwords
  - PDF SMR parsing for auto-populating KP catalogs
  - Daily schedule image generation (PNG via Pillow) and auto-publish
  - Role-based access control (7 roles) with session-cookie auth
  - Unified color system: roleConfig.js, statusConfig.js (Stage 6.4)
  - Framer-motion animations: modals, kanban, page transitions (Stage 6.5)
  - Desktop/mobile responsive adaptation (Stage 6.6)
  - PWA: installable app, service worker, push notifications (Stage 6.7 + v2.2)
  - Employee status system: vacation/sick leave tracking (v2.2)
  - FileViewerModal: PDF/image viewer with zoom and rotation (v2.2)
  - Comprehensive Guide (14 sections) and Updates changelog (Stage 6.8)
  - Complete action logging: 70+ log points with retention setting (Stage 6.9)
  - AI support chat with rate limiting (v2.2 + v2.3)
  - Full security audit: session auth on all ~120 endpoints (v2.3)

Stack: FastAPI + SQLite (aiosqlite, WAL) | React 19 + Vite 7 + Tailwind 3 + framer-motion | aiogram 3.27 (TG) + maxapi 1.0 (MAX)

--- CORE FLOW ---

  1. AUTH: User opens Telegram/MAX -> link code generated -> auth router
     validates -> session created (HttpOnly cookie).
     [SECURITY: HMAC verification for TMA (C-02), one-time codes for MAX
     (C-03), secrets.compare_digest for all token comparisons (M-08)]
     Account linking (Stage 5B): FIO auto-match detects same person on
     both platforms, admin can force-link in profile. Merged accounts
     share role, orders, and team memberships.

  2. CREATE ORDER: Foreman opens Kanban UI -> CreateAppModal -> selects
     object, team, equipment, date, time slot -> applications router
     validates resource availability (time-overlap detection) -> stores
     order as "pending".
     [SECURITY: Depends(get_current_user) on all endpoints, ownership
     checks for edit/delete (C-01)]

  3. REVIEW: Admin opens Review page -> approves/rejects -> bot
     notifications sent to foreman via Telegram and/or MAX. Rejection
     requires reason.
     [SECURITY: Depends(require_office) for review actions]

  4. PUBLISH: Approved orders -> schedule_generator renders PNG
     (Excel-like daily table with teams, equipment, sites) -> bots post
     image to Telegram group and MAX group chat. APScheduler handles
     auto-publish.

  5. EXECUTION: Workers view assignments via bot or web. Equipment
     exchange (Stage 5A) allows foremen to swap equipment between active
     orders. Time-based free-slot system prevents double-booking.
     Employee status (vacation/sick) tracked in schedule (v2.2).

  6. COST TRACKING: KP cost estimation workflow per order and per site.
     KP catalog uploadable via Excel (superadmin only). PDF SMR parser
     auto-extracts works. Mass export to Excel for reporting.
     [SECURITY: Depends(require_superadmin) for catalog upload (C-10)]

  7. FILES: Upload with safe_filename + UUID prefix + size/extension
     check + path traversal guard. Serve via /api/files/{id}/download
     with auth + ownership. FileViewerModal for PDF/image preview (v2.2).
     [SECURITY: .resolve().relative_to(UPLOADS_ROOT) on every path (H-08)]

  8. DASHBOARD: Aggregates daily stats (total/approved/rejected orders,
     top equipment usage, foremen activity). Cron jobs handle day
     start/end transitions, equipment timeout checks, and reminders.
     [SECURITY: Depends(get_current_user), PII stripped for non-office
     (H-04), cron endpoints behind HMAC-verified CRON_SECRET (H-10)]

--- ROLES ---

  superadmin > boss > moderator > foreman > brigadier > worker > driver

  superadmin  -- full system access, user management, settings, backups, logs
  boss        -- all except server logs, user management, broadcasts
  moderator   -- approve/reject orders, manage users/teams/equipment/objects
  foreman     -- create/edit own orders, manage own team, fill SMR reports
  brigadier   -- view orders, fill SMR reports
  worker      -- view own assignments, free brigade
  driver      -- view own assignments, mark equipment free
