--- APP OVERVIEW ---

A construction workforce management platform built around a Kanban-style order
(zayavka) system. Foremen create work orders assigning teams, equipment, and
schedules to construction sites. Admins review/approve orders, which are then
auto-published as visual schedule images to Telegram and MAX group chats.

Key capabilities:
  - Kanban order workflow (pending -> approved -> published -> in_progress -> completed)
  - Time-based equipment scheduling with free-slot detection
  - Equipment exchange between foremen (Stage 5A)
  - Account linking: merge Telegram <-> MAX identities (Stage 5B)
  - Cost estimation (KP) per order and per construction site
  - Team/brigade management with invite codes and passwords
  - PDF SMR parsing for auto-populating KP catalogs
  - Daily schedule image generation (PNG via Pillow) and auto-publish
  - Role-based access control (7 roles)

Stack: FastAPI + SQLite (aiosqlite, WAL) | React + Vite + Tailwind | aiogram (TG) + maxapi (MAX)

--- CORE FLOW ---

  1. AUTH: User opens Telegram/MAX -> link code generated -> auth router
     validates -> session created. Account linking (Stage 5B): FIO auto-match
     detects same person on both platforms, admin can force-link in profile.
     Merged accounts share role, orders, and team memberships.

  2. CREATE ORDER: Foreman opens Kanban UI -> CreateAppModal -> selects object,
     team, equipment, date, time slot -> applications router validates resource
     availability (time-overlap detection) -> stores order as "pending".

  3. REVIEW: Admin opens Review page -> approves/rejects -> bot notifications
     sent to foreman via Telegram and/or MAX. Rejection requires reason.

  4. PUBLISH: Approved orders -> schedule_generator renders PNG (Excel-like
     daily table with teams, equipment, sites) -> bots post image to
     Telegram group and MAX group chat. APScheduler handles auto-publish.

  5. EXECUTION: Workers view assignments via bot or web. Equipment exchange
     (Stage 5A) allows foremen to swap equipment between active orders.
     Time-based free-slot system prevents double-booking.

  6. COST TRACKING: KP cost estimation workflow per order and per site.
     KP catalog uploadable via Excel. PDF SMR parser auto-extracts works.
     Mass export to Excel for reporting.

  7. DASHBOARD: Aggregates daily stats (total/approved/rejected orders,
     top equipment usage, foremen activity). Cron jobs handle day start/end
     transitions, equipment timeout checks, and reminders.

--- ROLES ---

  superadmin > boss > admin > foreman > master > worker > viewer

  superadmin  -- full system access, user management, settings, backups
  boss        -- read-all access, no editing restrictions
  admin       -- approve/reject orders, manage users/teams/equipment
  foreman     -- create/edit own orders, manage own team
  master      -- limited order creation
  worker      -- view own assignments
  viewer      -- read-only access

--- ARCHITECTURAL INSIGHTS ---

  1. GOD MODULE RISK: web/utils.py has high fan-out (11+ edges). It mixes
     image generation, font management, team enrichment, app publishing,
     and linked ID resolution. Consider domain-based splitting.

  2. LOW COHESION WARNING: The Database & Account Merge cluster (74 nodes,
     cohesion 0.03) suggests DatabaseManager aggregates too many concerns.
     The mixin pattern helps, but the manager itself is a single point of
     coupling.

  3. WELL-ISOLATED FEATURES: Equipment Exchange (0.29), PDF Parser (0.31),
     MAX Bot (0.39), Auth & Linking (0.24) are cleanly separated -- good
     modular boundaries.

  4. 112 ISOLATED NODES: Many functions have <=1 connection, suggesting
     either missing documentation or truly standalone utilities. Consider
     adding integration tests for these to verify they are still in use.

  5. DUAL-BOT ARCHITECTURE: main.py (Telegram) and main_max.py (MAX) share
     the same database but have independent event loops. Account linking
     (Stage 5B) bridges identities across both platforms via linked_user_id.
